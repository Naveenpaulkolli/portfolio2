Artifact 5 – Android Malware Detection: Model Validation & Ethics

Target Audience: Entry-level AI/ML hiring managers and course instructors evaluating my ability to design trustworthy ML workflows for security use cases.

Introduction

This artifact presents a validation and verification framework for detecting Android malware using classic ML techniques. It focuses on how I would clean and prepare features, select models, validate performance, and address ethics and ongoing monitoring—skills that matter in real-world security analytics.

Description

Using the public Android Malware Dataset for Machine Learning (Yerima, 2018), I outline a practical plan to turn app-level features (permissions, API/call patterns) into a reproducible pipeline. Rather than claiming specific accuracy, this artifact emphasizes defensible process: data readiness, model choice rationale, fair evaluation, and post-deployment safeguards.

Objective

  • Design a transparent validation approach that balances accuracy with fairness and reliability.
  • Document how to monitor drift and retrain as new malware families appear.
  • Demonstrate ethical awareness (privacy, bias, responsible use) in a security context.

Process

  • Data Readiness: Inspect missing values, duplicate apps, and label integrity; standardize feature types; remove leakage columns (e.g., post-label artifacts).
  • Feature Handling: One-hot/boolean encoding for permissions; frequency caps for rare permissions; optional PCA for very high-dimensional vectors.
  • Class Imbalance: Compare class_weight vs. resampling (SMOTE). Start with class weights to avoid synthetic artifacts; document tradeoffs.
  • Model Candidates: Logistic Regression (baseline, interpretable), Random Forest (nonlinear patterns, robust), Linear SVM (high-dimensional sparse signals).
  • Validation: Stratified train/validation/test split; 5-fold stratified CV; track seeds for reproducibility. Guardrail on metrics leakage.
  • Metrics: Precision/Recall, F1 (macro), ROC-AUC/PR-AUC; confusion matrix by class; threshold tuning based on business risk (false-negatives vs. false-positives).
  • Error Analysis: Review top false-negatives (missed malware) and false-positives (benign apps flagged); examine permission patterns and SHAP-style feature attributions.
  • Post-Deployment: Drift checks (population stability index on key features), monthly performance re-eval, safe rollback, and a retraining playbook on fresh samples.
  • Ethics & Privacy: Use only public, de-identified features; document limitations and avoid over-claiming “safety.” Provide appeal/override path for flagged apps.

Tools & Technologies Used

  • Python with pandas, scikit-learn (StratifiedKFold, class_weight, pipelines).
  • Visualization for diagnostics: confusion matrix, PR curves, drift charts.
  • Reproducibility: requirements.txt, fixed random seeds, simple experiment log.

Value Proposition

Unique Value: Instead of only chasing accuracy, this artifact shows I can design a trustworthy workflow—clear validation, risk-aware thresholds, and ongoing monitoring for a changing threat landscape.

Relevance: Security and compliance teams need explainable, maintainable models. This plan fits enterprise expectations for governance and model lifecycle care.

References

  • Yerima, S. (2018). Android malware dataset for machine learning 2. figshare. https://doi.org/10.6084/m9.figshare.5854653.v1
  • Yerima, S., & Sezer, S. (2017). Machine Learning for Android Malware Detection. IEEE (paper referenced by dataset record).
